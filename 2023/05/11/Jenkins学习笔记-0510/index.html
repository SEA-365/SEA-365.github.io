Jenkins学习-流水线

Jenkins流水线

  • 官方文档：Jenkins 流水线是一套插件，它支持实现和集成持续交付流水线到 Jenkins。

    • 不是很明白

  • gpt:一种基于Jenkins构建的持续集成和交付（CI/CD）解决方案。它 允许 开发者 通过 定义一系列的步骤 来 自动化构建、测试、部署和监控应用程序 的流程。

    • 流水线是由一系列步骤组成的工作流程，这些步骤可以在Jenkins中以代码的形式进行定义和管理。
    • 通过在Jenkins中创建流水线，开发者可以将构建、测试、部署和监控等任务自动化，从而提高应用程序的质量和可靠性，并缩短发布周期。

创建和定义流水线

使用Jenkinsfile文件来定义流水线是Jenkins的最佳实践之一。

  • 声明式和脚本式流水线都是 DSL(Domain-specific language 领域特定语言) 语言，用来描述软件交付流水线的一部分。

    • 二者都是Jenkins Pipeline的DSL语言，用于编写Jenkins流水线中的步骤和操作。
    • 声明式：声明式流水线是一种更为简单和易于使用的语言，旨在为用户提供一种简单的方式来编写Jenkins流水线。声明式流水线提供了一组预定义的指令，可以让用户轻松地定义流水线的结构和逻辑。声明式流水线更加可读性强，易于理解，并支持自动代码生成和可视化编辑器等功能。
    • 脚本式：脚本式流水线是更为灵活和强大的语言，旨在为有经验的用户提供更大的自由度和控制力。脚本式流水线允许用户使用Groovy语言来编写完全自定义的脚本，从而实现复杂的逻辑和自定义操作。脚本式流水线的优点是可扩展性强，用户可以通过编写自定义的Groovy脚本来实现特定的需求和功能。

分析Jenkinsfile文件

  • agent指令：这是必需的一个部分，指示 Jenkins 为流水线分配一个执行器和工作区；

    • 如果没有agent指令，声明式流水线不仅无效，它也不可能完成任何工作！
    • 默认情况下，agent 指令确保源代码仓库被检出并在后续阶段的步骤中可被使用。

  • 一个合法的声明式流水线还需要stages 指令和steps指令，因为它们指示 Jenkins 要执行什么，在哪个阶段执行。

构建阶段

  • 对于许多项目来说，流水线“工作”的开始就是“构建”阶段。

  • Jenkinsfile 文件不能替代现有的构建工具，如 GNU/Make、Maven、Gradle 等，而应视其为一个将项目的开发生命周期的多个阶段（构建、测试、部署等）绑定在一起的粘合层。

    • Jenkins 有许多插件可以用于调用几乎所有常用的构建工具；

    • sh 步骤调用 make 命令，只有命令返回的状态码为零时才会继续。任何非零的返回码都将使流水线失败。

    • archiveArtifacts 捕获符合模式（**/target/*.jar）匹配的交付件并将其保存到 Jenkins master 节点以供后续获取。

测试阶段

  • 运行自动化测试是任何成功的持续交付过程的重要组成部分；
  • Jenkins通过插件通提供了许多测试记录，报告和可视化工具；
  • 最基本的，当测试失败时，让 Jenkins 记录这些失败以供汇报以及在 web UI 中可视化是很有用的。

部署阶段

  • 部署可以隐含许多步骤，这取决于项目或组织的要求，并且可能是从发布构建的交付件到 Artifactory 服务器，到将代码推送到生产系统的任何东西。
  • 流水线访问 currentBuild.result 变量确定是否有任何测试的失败。在这种情况下，值为UNSTABLE。
  • 从本质上讲，“Deploy（部署）” 阶段只有在之前的阶段都成功完成后才会进行，否则流水线会提前退出。
  • 假设在基本三阶段持续交付的示例中，Jenkins 流水线中所有的操作都执行成功，那么每次流水线的成功运行都会在 Jenkins 中存档相关的交付件、上面报告的测试结果以及所有控制台输出。
